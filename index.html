Brandeburg Gate

A Photograph by John Waterman

Taken From Google Arts and Culture

Why do we choose this item?

As we needed at least an item to represent Berlin, we thought that one of the most important cultural sites (and symbols) in Berlin is the Brandeburg Gate. This element is quite important in our model because what we developed for the Berlin Wall, can be reused (with, of course, some adjustemts) for other cultural sites in the city. With this photograph, that is the first one, we want to contextualize our project. In some sense we want the user enter imaginig the setting (in this case the city) where the Berliner Mauer was built.

What about Metadata and Standards?

Searching in the page's Source Code we found that the page has some metadatas using the "OG" standard. This is interesting because even if the metadata are refereed to the whole page, the latter contains just the image (and related inforamtion), so metadata describe the item. Then, Google Cultural Institute uses its own "Standard"(in Google Cultural Platform Help page you can find the "Metadata Fields" and "Prepare a metadata file (Google Sheets)"). It seems to be similar (or at least modelled on) Dublin Core.

    Google Metadata Standard (General CSV)
  • title: A short title for the item. (Required)
  • creator: Creator of item such as the author, painter, or photographer (Recommended)
  • dateCreated: Date an item was created, such as when a painting was painted. (Recommended)
  • provenance: Description of provenance of object. (Optional)
  • subject (Subject Keywords): Free text field for keywords representing topic of item. (Optional)
Issues
  • In the photograph there is a signal: What is written on the signal is strictly related to West and East Berlin split (main point of our project). Taking this case as an example (but we will find other similar aspects in the following items) and thinking about our Conceptual Model, we can imagine to develop a particultar metadata tag that contains everything is written in an image (specifying also some other characteristics).
  • Some metadata fields considered required by Google Standard are not fulfilled (ex: itemid)

Murs

A Video by TV3 Televisió de Catalunya (TVC)

Taken From EUscreen portal

Why do we choose this item?

“30 Minuts visited Mexico, the United States, Morroco, Berlin, Palestine and Israel to see how the lives of the people living there have been affected–often tragically– by the building of walls to separate people.” The video is so representing the concept of Segregation, taking into account the Belin Wall's facts. This item can be seen as a nice way to know more about the Wall through the concept of Segregation and so, trying to let the user start to think about the Wall approching first the "conceptual" and profound meaning at it had (and still has) in society.

What about Metadata and Standards?

EUScreen metadata are modelled through EBUCORE Standard (Here you can find some information about Ebucore and Europeana). Ebucore standard is ment to "provide the framework for descriptive and technical metadata for use in service orientated architectures and audiovisual ontologies for semantic web and linked data developments" (more info). Ebucore is modelled reusing existing standards schema-org, Dublin Core and FOAF.

    Item metadata compared with EBUCORE documentation:
  • OriginalTitle: To provide the original title attributed to the media resource e.g. in its original language.
  • dc:contributor (provider): An Agent who has participated in any phase of management of a BusinessObject or Resource
  • Creation date/time (production year): The date of creation of the Resource or BusinessObject.
  • dc:language: A Language used in the BusinessObject or Resource
  • Summary: To provide a summary (subProperty of Description)
  • Parent season / series (This item is part of a series/collection ):The Episode of a Series or a Season. (+ SeasonNumber, EpisodeNumber)
  • dc:publisher (publisher/broadcaster): An Agent involved in the distribution of content.
  • Broadcast date: The date when the Resource was first broadcast publicly on television or radio or via streaming.
  • Country: To provide the country name and or country code.
  • OriginalLanguage: The original language in which the BusinessObject or Resource has been created and released. This is provided as free text in an annotation label or as an identifier pointing to a term in a classification scheme
  • DubbedLanguage: Dubbed language
  • Coverage (Geographical Coverage): A property to identify the Events and or Locations, all real or fictional, covered by the BusinessObject
  • dc:contributor: An Agent who has participated in any phase of management of a BusinessObject or Resource.
  • Media type: To provide additional information on the type of media
  • Item: An item e.g. newsItem or sportItem -- All the metadata provided from this point are referred to the item (published duration, colour, sound, aspect ratio )
  • genre: To define a Genre/category associated to the BusinesssObject
  • primarytopic (topic): The main subject of interest. (subclass of description)
  • keyword: To associate a concept, descriptive phrase or Keyword that specifies the topic of the BusinessObject or Annotation.

Analysing EBUCORE we found that it is really specific in classifying techincal audiovisual elements (e.g.: subtitle, but also aspect ratio, maximum bitrate, Max momentary loudness) and it has also a really accurate specification of description (e.g: summary, primary topic).

Our Theoretical Model

This model is able to describe the chosen items replying to the questions:
  • WHO – people. How to describe people? Which kind of information?
  • WHERE – places. Which kind of information I have to represent about locations?
  • WHEN – dates. In which format I need to express the notion of time?
  • WHAT – subjects/concepts. What is the main content of my object?
WHO

At this point we noticed that we need to treat people in double aspect:

  • First we need to provide some anagrafical information about the acting character; We decided to use FOAF oppure RDAGR2 (DublinCore ti amo)
  • then how this person is involved with the Berlin Wall history in this specific case.
    So, we thought that the realtion between The Berlin Wall as monument and Peter Fechter as a person is that Fechter was a sort of living example/exponent of an important historical event. This concept is foundamental because our aim is to present items related to Peter Fachter as a concrete example/metaphor as "portatore" of a concept, in this case "Segregation".
    For this purpose we used LODE, used for the descrition of events. ONTOLOGIA DELLA BBC BELLISSIMA: BCC STANDRD (proprietà: Date of birth/death, place of date/birth, event palce/theme/person)

WHERE

At this point we noticed that we need to treat places under many aspects in order to have a broad vision in our model:

  • GEOLOCALIZATION: Places can be seen as points in space. For this reason we want to convey to the user the coordinates of the place of interest. In order to do so we will use wgs84_pos. This standard has an additional useful feature in our model: we can provide a CSV file to indicate a set of points in the space, so to present lines and shapes. This is the specific case of the Berlin Wall building (a set of lines).
  • PLACE AS A CULTURAL SPOT: Our concept model is also based on the idea that the places we present are also places of cultural interest. For this reason we want to provide some useful information about it. In order to do this we will use schema and dbo.
  • POLITICAL LOCALIZATION: To connect this specific place to the city which it belongs we will add some information about the actual city. In order to do this we will use schema providing the specificity of the specification with dbo.
  • HISTORICAL REGION: Considering the fact that we are working on a model which has a strong historical connotation, and considering that we are talking about Berlin in our case-study, we want to add some additional information about the "historical" region, where this place was contained (East Berlin). In order to do so we will use schema joined with owl-time.

WHEN

LODE:
LIDO: Its strength lies in its ability to support the full range of descriptive information about museum objects. It can be used for all kinds of object, e.g. art, cultural, technology and natural science. It is made up of a nested set of ‘wrapper’ and ‘set’ elements which structure records in culturally significant ways. An important part of its design is the concept of events taken from the CIDOC CRM.
